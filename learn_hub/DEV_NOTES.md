# LearnHub — Development Complexities & Lessons Learned

## Overview

This document captures the real challenges encountered while building the LMS frontend in Jac. It serves as a reference for future Jac full-stack development.

---

## 1. Incorrect API Call Pattern

**What happened:**
The initial frontend used `__jacCallFunction` directly — an internal runtime function that is not part of the public developer API.

```jac
// WRONG
import from "@jac/runtime" { __jacCallFunction };
let result = await __jacCallFunction("get_courses", {});
```

**Why it's wrong:**
`__jacCallFunction` is a low-level bridge generated by the compiler. Calling it manually bypasses the type system, provides no IDE support, and breaks if internal runtime behaviour changes.

**The fix:**
Use `sv import` to import server-side `def:pub` functions. The compiler generates the HTTP bridge automatically.

```jac
sv import from .main { get_courses, create_course, delete_course };

// Then call with:
let result = await get_courses();
```

## 2. Manual `useState` / `_setter` Hacks

**What happened:**
Attempted to manually call `useState` and used `_setter["fn"]` patterns to manage reactive state, mimicking raw React patterns.

```jac
// WRONG
import from "@jac/runtime" { useState };
let [count, setCount] = useState(0);
_setter["setCount"](count + 1);  // manual hack
```

**Why it's wrong:**
Jac compiles `has` variable declarations inside components directly to `useState`. Manually importing and calling `useState` bypasses this, produces duplicate state declarations, and the `_setter` hack is fragile and undocumented.

**The fix:**
Use the `has` keyword for all reactive state inside components.

```jac
obj Counter {
    has count: int = 0;

    can render -> JsxElement {
        return <button onClick={() -> { count = count + 1; }}>
            {count}
        </button>;
    }
}
```

`count = count + 1;` compiles to `setCount(count + 1)` automatically.

## 3. Total Frontend-Backend Disconnection

**What happened:**
The original frontend had zero working connectivity to the backend despite the backend being fully functional and tested.

**Root causes (combined):**
- Using `__jacCallFunction` directly with incorrect argument shapes
- Missing `sv import` declarations entirely
- Manual hook hacks conflicting with Jac's compiled output
- Wrong mental model: treating Jac frontend as raw React/JS

**The fix:**
Complete rewrite from scratch after studying Jac documentation and working examples (the todo app in the repo). Connectivity worked on the first run of the rewritten version.

## 4. `sv import` Uncertainty: Functions vs Walkers

**What happened:**
All working examples in the codebase use *walkers* with `sv import`. The backend here uses `def:pub` functions. It was unclear whether `sv import` would work for functions.

**Working examples showed (walkers):**
```jac
sv import from .main { TodoWalker };
let result = root spawn TodoWalker();
let data = result.reports[0];
```

**Needed to use (functions):**
```jac
sv import from .main { get_courses };
let data = await get_courses();
```

**Resolution:**
`sv import` works for both walkers and `def:pub` functions. After the build, all functions appeared in the HTML's `endpointEffects` JSON block and compiled JS correctly generated `__jacCallFunction` wrappers for each. The difference is call syntax: walkers use `root spawn WalkerName()`, functions use `await func_name()`.

## 5. Type Checker False Positives (~80+ Warnings)

**What happened:**
The IDE reported ~80 errors and warnings in `lms_fe.cl.jac` after writing it, making it look broken.

**Examples of false positives:**

| Warning | Reason it's a false positive |
|---------|-------------------------------|
| `.trim()` not found on `str` | JS method, Python type system uses `str.strip()` |
| `.length` not found on `list` | JS property, Python uses `len()` |
| `parseInt` undefined | JS global, not in Python scope |
| `lambda e: any` conflicts with `any()` | Python's `any()` builtin shadows the type hint |
| `has x` "defined but never used" | Variables used in JSX but checker doesn't track JSX |

**Why they appear:**
Jac's type checker validates against Python's type system even for `.cl.jac` client files. Client code compiles to JavaScript, so JS APIs (`trim`, `length`, browser globals) are valid at runtime but invisible to the Python-based checker.

**Resolution:**
These are not real errors. The todo app in this same repo would produce identical warnings. The code compiled and ran successfully despite all warnings.

## 6. Jac Syntax Habits vs Python Habits

**What happened:**
AI-generated code (and initial developer assumptions) repeatedly defaulted to Python syntax, which is invalid in Jac.

**Key syntax differences that caused confusion:**

| Python habit | Correct Jac |
|-------------|-------------|
| No semicolons | Semicolons required on ALL statements |
| Indentation for blocks | Braces `{ }` for all blocks |
| `class Foo:` | `obj Foo { }` |
| `def __init__(self):` | `has field: type = default;` |
| `from x import y` | `import from x { y }` |
| `useEffect(fn, [])` | `async can with entry { }` |
| `if __name__ == "__main__":` | `with entry { }` |

## 7. Null/Falsy API Responses

**What happened:**
API calls for empty lists (e.g., a course with no lessons yet) returned falsy values instead of `[]`, causing crashes when the component tried to iterate over the result.

**Example crash scenario:**
```jac
has lessons: list = [];

async can with entry {
    let result = await get_lessons(course_id);
    lessons = result;  // result could be null/undefined on empty
}

// Later in render:
// lessons.map(...)  →  CRASH: cannot map over null
```

**The fix:**
Always apply a null guard when assigning API results:
```jac
lessons = result if result else [];
```

## 8. File Convention: `.cl.jac` vs `cl { }` Block

**What happened:**
Confusion over two ways to write client-side code in Jac.

**The two approaches:**
1. **`.cl.jac` files** — Entire file is automatically client-side. No wrapper needed.
2. **`cl { }` block** — Inline client code within a regular `.jac` file.

**Resolution:**
For a dedicated frontend file, use `.cl.jac`. The `cl { }` block is for mixed files where some code is server-side and some is client-side.

## 9. Routing: File-Based vs Manual

**What happened:**
Jac supports two routing strategies and the documentation for each overlaps, making it unclear which to use for this project.

**File-based routing** (`pages/` directory):
- Create files like `pages/login.jac`, `pages/dashboard.jac`
- Routes are inferred from file names
- Better for large apps with many pages

**Manual routing** (used in this project):
```jac
import from "@jac/runtime" { Router, Routes, Route, Link, Outlet };

def:pub App() -> JsxElement {
    return <Router>
        <Routes>
            <Route path="/login" element={<LoginPage />} />
            <Route path="/" element={<ProtectedLayout />}>
                <Route index element={<Dashboard />} />
            </Route>
        </Routes>
    </Router>;
}
```

**Decision:** Manual routing was used for simplicity in a single-file frontend.

## 10. Auth Function Return Types

**What happened:**
Incorrect assumptions about what `jacLogin`, `jacSignup`, etc. return caused incorrect conditional logic.

**Actual return types:**

| Function | Returns | Notes |
|----------|---------|-------|
| `jacLogin(username, password)` | `bool` | `true` on success |
| `jacSignup(username, password)` | `dict` | Has `success: bool` key |
| `jacIsLoggedIn()` | `bool` | Synchronous check |
| `jacLogout()` | `void` | — |

**Example of wrong assumption:**
```jac
// WRONG: treating jacSignup return as bool
if await jacSignup(username, password) { ... }

// CORRECT
let result = await jacSignup(username, password);
if result["success"] { ... }
```

---

## Key Takeaway

The root cause of most issues was **relying on assumptions** (from Python, React, or prior AI output) rather than reading the actual Jac documentation and studying working examples in the repository.

**The reliable workflow for Jac frontend development:**
1. Read `jac://guide/pitfalls` before writing any code
2. Read `jac://guide/patterns` for idiomatic examples
3. Study a working full-stack example in the repo (e.g., the todo app)
4. Write code, then validate with `mcp__jac-mcp__validate_jac`
5. Fix any errors using `mcp__jac-mcp__explain_error` before testing

Once proper patterns were followed, the rewritten frontend compiled and connected to the backend on the **first run**.
